# 冷库工作台功能映射表

这份文档先不改代码，只把“冷库工人实际要做的事”和 InvenTree 现有能力对起来。我的建议是第一版先做一个并行的“冷库工作台”，复用库存底层能力，不直接重做整套 InvenTree 页面。

## 总体判断

第一版推荐走“库存直操”路线：

- 不先启用完整采购单、销售单、生产单流程。
- 普通工人只看到少量大按钮和中文表单。
- 后台仍然用 InvenTree 的 Part、StockItem、StockLocation、Stock API 保存数据。
- 管理员仍可进入原 InvenTree 页面查明细、改配置、做高级处理。

原因是冷库现场最常见的是“来了多少、放哪了、拿走多少、还剩多少、过期报损”，这和现有 StockItem/StockLocation 能力更贴近；采购/销售/生产流程对一线人员太重。

## 功能对应关系

| 冷库说法 | 工人看到的入口 | 现有 InvenTree 能力 | 建议第一版做法 |
| --- | --- | --- | --- |
| 新货入库 | 新货入库 | Part 建档、StockItem 创建、stock/add | 如果货品不存在，先简化建档再创建库存；如果已存在，直接加库存 |
| 出库扣数 | 出库扣数 | stock/remove | 选择货品和库位，输入数量，原因写入备注 |
| 查库存 | 查库存 | StockItem 列表、Part 库存汇总 | 只显示品名、数量、单位、库位、批次、到期日期 |
| 改库位 | 改库位 | stock/transfer | 从原库位转到新库位，不暴露包装、序列号等复杂字段 |
| 盘点改数 | 盘点改数 | stock/count、StockItem.stocktake | 显示系统数量，输入实际数量，差异原因写备注 |
| 报损/过期 | 报损/过期处理 | stock/remove、库存状态、备注 | 第一版按“扣库存 + 报损原因”处理，后续再细分状态 |
| 新建库位 | 新增冷库/货架 | StockLocation | 管理员或班组长使用，不建议普通工人随便建 |
| 货品资料 | 货品档案 | Part | 只保留品名、单位、默认库位、保质期、备注等字段 |

## 现有代码线索

- 前端库存表单：`src/frontend/src/forms/StockForms.tsx`
  - `useStockFields` 已包含货品、库位、数量、批次、到期日期等字段。
  - `useCreateStockItem` 可创建库存项。
  - `stockAddFields`、`stockRemoveFields`、`stockTransferFields`、`stockCountFields` 可作为后续简化表单的参考。
- 后端库存接口：`src/backend/InvenTree/stock/api.py`
  - `stock/add/`：加库存。
  - `stock/remove/`：扣库存。
  - `stock/transfer/`：转库位。
  - `stock/count/`：盘点改数。
  - `stock_item_list`：创建或查询库存项。
- 后端库存模型：`src/backend/InvenTree/stock/models.py`
  - `StockItem.stocktake` 支持盘点数量、用户、备注、库位变化和历史记录。
  - `StockLocation` 支持树状库位，适合“冷库 -> 区域 -> 货架/位置”。
- 货品建档：`src/frontend/src/forms/PartForms.tsx` 和 `src/backend/InvenTree/part/serializers.py`
  - Part 创建时已有基础字段。
  - 后端 `PartSerializer.create` 支持 `initial_stock`，可以在建档时带初始库存，但批次、到期日这类冷库常用字段后面要再确认是否需要拆成“建货品 + 建库存项”两步。

## 第一版页面建议

普通工人首页只放 6 个入口：

1. 新货入库
2. 出库扣数
3. 查库存
4. 改库位
5. 盘点改数
6. 报损/过期

每个表单都按“少填、能看懂、不出英文”的原则做。能自动带出的不要让工人填，比如当前日期、操作人、默认库位、单位。

## 字段精简建议

### 新货入库

必填：

- 货品名称
- 数量
- 单位
- 冷库/库位

可选：

- 到货日期
- 到期日期
- 批次
- 供应商
- 备注

### 出库扣数

必填：

- 货品名称
- 出库数量
- 冷库/库位

可选：

- 出库原因
- 领用人/去向
- 备注

### 查库存

搜索条件：

- 货品名称
- 冷库/库位
- 批次
- 到期状态

列表列名：

- 货品
- 数量
- 单位
- 冷库/库位
- 批次
- 到期日期

### 盘点改数

必填：

- 货品名称
- 冷库/库位
- 实际数量

系统自动显示：

- 系统数量
- 差异数量
- 操作人
- 操作时间

### 报损/过期

必填：

- 货品名称
- 数量
- 冷库/库位
- 原因

原因第一版可先固定为：

- 过期
- 破损
- 变质
- 盘亏
- 其他

## 权限建议

| 角色 | 能看到什么 | 能做什么 |
| --- | --- | --- |
| 普通工人 | 冷库工作台 | 入库、出库、查库存、转库位、盘点、报损 |
| 班组长 | 冷库工作台 + 少量管理入口 | 新建库位、审核异常、看操作记录 |
| 管理员 | 原 InvenTree 全部功能 | 系统配置、用户权限、完整库存管理 |

这里后面要重点处理权限。否则工人虽然看的是简化页面，但如果还能进原始页面，仍然可能误操作复杂功能。

## 暂时不建议第一版做的事

- 不建议把生产、采购、销售整套页面都汉化重做。
- 不建议一开始就强制走采购单/销售单审批流。
- 不建议普通工人接触 IPN、BOM、Build、Assembly、Supplier Part 这些概念。
- 不建议先做扫码，因为你已经说明现场没有条码，只能手动输入。

## 需要你补充确认的问题

1. 货品有没有保质期或到期日期？是否必须按批次管理？
2. 出库时需要记录“领用人/客户/去向”吗？
3. 报损是否需要班组长或管理员审核？
4. 库位层级大概怎么叫：冷库、区域、货架、层、位置，还是更简单？
5. 普通工人是否允许新建货品，还是只能管理员先建好货品？
6. 是否允许库存数量改成负数？我的建议是不允许。
