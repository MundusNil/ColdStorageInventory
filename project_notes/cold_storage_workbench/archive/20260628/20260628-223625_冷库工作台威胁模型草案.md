# 冷库工作台威胁模型草案

本文档是草案，不是最终安全报告。正式威胁模型需要等用户确认部署方式、角色权限和现场流程后再定稿。

## 当前假设

- 系统部署在本地或内网，主要给冷库工作人员和管理员使用。
- 普通工人不应该看到完整 InvenTree 管理页面。
- 普通工人可以做入库、出库、查库存、改库位、盘点、报损，但具体是否需要审核未确认。
- 管理员仍可使用原 InvenTree 全功能。
- 现场没有条码，只能手动输入。

## 主要资产

| 资产 | 为什么重要 |
| --- | --- |
| 用户账号和会话 | 被盗后可直接改库存或看业务数据 |
| 库存数量 | 是冷库系统最核心的数据，错误会影响发货和盘点 |
| 库位信息 | 决定货品实际存放位置，错了会找不到货 |
| 操作记录 | 后续追责、盘点差异分析需要依赖 |
| 管理员权限 | 可以配置系统、改用户、改全部库存 |

## 信任边界

| 边界 | 数据 | 风险 |
| --- | --- | --- |
| 浏览器 -> 前端页面 | 用户输入、会话状态 | 表单误导、权限只靠前端隐藏 |
| 前端页面 -> API | 入库/出库/盘点请求 | 越权修改库存、重复提交 |
| API -> 数据库 | 库存和用户数据 | 写入错误、审计记录缺失 |
| 普通工人 -> 管理员功能 | 导航和路由访问 | 只隐藏页面但未限制接口权限 |

## 初步威胁

| ID | 威胁 | 影响 | 初步缓解 |
| --- | --- | --- | --- |
| TM-001 | 普通工人通过原路由访问复杂页面并误改库存 | 库存数量或库位错误 | 后续按角色控制入口和后端权限 |
| TM-002 | 出库/盘点按钮重复提交 | 库存被多扣或盘点数量异常 | 提交中禁用按钮，后端记录幂等或审计 |
| TM-003 | 手动输入选错货品 | 错货入库/出库 | 搜索结果显示库位、批次、到期日，关键操作二次确认 |
| TM-004 | 管理员账号用于日常操作 | 高权限误操作 | 保留超级用户提醒，普通工作流使用低权限账号 |
| TM-005 | 登录密码出现在 URL 参数或日志中 | 凭据泄露 | 移除 URL 密码登录，异常日志避免记录敏感查询参数 |
| TM-006 | 只读库存表意外暴露库存调整动作 | 普通工人误改库存 | 冷库工作台传入 `readOnly`，隐藏选择、标签、报表和库存调整动作 |

## 需要用户确认

1. 这个系统最终是在单机、本地局域网，还是公网可访问？
2. 普通工人是否允许直接改库存数量，还是必须班组长审核？
3. 冷库系统会存客户、供应商、价格等敏感数据吗，还是只存货品和库存？
